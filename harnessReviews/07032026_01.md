## Findings And Fixes

### Fixed: Secret Disclosure Through Config Repr

Finding:

- `McpServerConfig.env` could appear in dataclass repr output after `.env` expansion.
- This created an accidental secret disclosure risk in debug logs, tracebacks, REPL output, and support reports.

Fix:

- Marked `McpServerConfig.env` with `repr=False`.
- Added regression coverage verifying loaded secret values do not appear in `repr(cfg)`.

### Fixed: Unsafe Boolean And Numeric Config Coercion

Finding:

- Config values used Python coercion such as `bool(value)`, which makes YAML strings like `"false"` truthy.
- Nonpositive limits and timeouts could be accepted until runtime failure.

Fix:

- Added explicit config validators for booleans, positive integers, positive floats, and nonnegative temperature.
- Added validation for MCP server names and startup timeout bounds.
- Added tests for string boolean rejection and nonpositive numeric rejection.

### Fixed: Runtime `.env` Parser Drift

Finding:

- Setup and runtime config loading did not parse `.env` the same way.
- Setup could produce values that later loaded incorrectly if credentials contained spaces, quotes, or comment characters.

Fix:

- Added deterministic `.env` quoting in setup.
- Rejected embedded newlines in credential values.
- Switched both setup reader and runtime loader to shell-style parsing.
- Added round-trip tests for quoted credentials.

### Fixed: Shell Verification Classifier Accepted Mutations

Finding:

- Commands such as `find . -type f -delete` and `sed -i ...` were treated as verification because the leading command looked read-only.
- This could allow a mutating shell command to clear the post-mutation verification gate.

Fix:

- Added mutating-flag detection for read-like shell commands.
- Added regression tests for `find -delete`, `find -exec`, and `sed -i` class behavior.

### Fixed: Shell Path Escape Blind Spot

Finding:

- Shell path screening caught absolute paths but not parent-relative escapes such as `../escaped`.

Fix:

- Extended shell token path detection to catch `..` and `../...` references.
- Added regression test proving `mkdir -p ../escaped` is blocked outside the workspace unless allowlisted.

### Fixed: MCP Stdio Stderr Deadlock Risk

Finding:

- MCP child processes had `stderr=PIPE`, but stderr was only read on certain failure paths.
- A noisy MCP server could fill stderr and block while the client waited on stdout.

Fix:

- Added a daemon stderr-drain thread with a bounded 50-line tail buffer.
- Timeout and process-exit errors now include recent stderr context.
- Added regression coverage for bounded stderr tail retention.

### Fixed: Bare Tool Command Quoting

Finding:

- Bare tool commands were parsed with `.split()`, so quoted paths and arguments were not preserved.

Fix:

- Switched bare command parsing to `shlex.split`.
- Added regression coverage for quoted paths.

### Fixed: CI Did Not Run Advertised Quality Gates

Finding:

- GitHub Actions installed only runtime dependencies and did not run Ruff or mypy, even though the project advertises those checks.

Fix:

- CI now installs `".[dev]"`.
- CI now runs `ruff check .` and `mypy localforge tests`.

### Fixed: Active Network Fetch Policy Was Implicit

Finding:

- Active `localforge.yaml` enabled network fetch but did not explicitly show private-network fetch policy.

Fix:

- Added `allow_private_network_fetch: false` to the active config.

## Remaining Risks

- MCP profiles still use `@latest`, which is convenient but not reproducible. Pinning should be added before commercial release.
- Docker Compose mounts `/var/run/docker.sock`; this is powerful host-level access. It is documented, but a safer profile-gated compose setup would reduce accidental exposure.
- `fetch_url` blocks private resolved addresses and redirects, but DNS rebinding remains a hard SSRF class to eliminate without a stricter resolver/transport design.
- The agent verification gate is heuristic for MCP tool names and shell commands. It is safer than before, but a declarative tool capability policy would be stronger.
- No OpenTelemetry export path exists yet; run JSONL is useful but not enough for fleet-level monitoring.
- No signed release/provenance workflow exists.
